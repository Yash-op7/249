# 1️⃣ Idempotency keys – the real purpose

Let’s start with a problem.

---

## The real-world problem

A client sends:

`POST /payments`

But:

- network times out
- client retries
- load balancer retries
- browser retries
- mobile app retries
    

Now your server receives **the same logical request twice**.

If you just use payment state:

`CREATE payment CREATE payment again`

💥 Double charge.

---

## Why payment ID is NOT enough

⭐ Payment ID exists **after** the request succeeds.

⭐ But idempotency is needed **before** you create anything.

⭐ You must detect duplicates _at request boundary_

---
## What is an idempotency key?

A client-generated unique key per logical operation:

`Idempotency-Key: 8f3a-92bc-...`

Same operation → same key  
Different operation → different key

---

## How idempotency actually works (precise flow)

### Example: Create payment

#### First request

```yaml
POST /payments
Headers:
  Idempotency-Key: abc123

Body:
  { amount: 1000, currency: "USD", direction: "IN" }

```

API:

1. Hash request body → `H1`
    
2. Look up `abc123` in `idempotency_keys`
    

Not found →

3. Start DB transaction
    
4. Create payment row
    
5. Store:
    

```css
idempotency_keys:
  key = abc123
  request_hash = H1
  response_json = { payment_id: ... }
  expires_at = now + 24h
```
3. Commit
    
4. Return response
    

---

#### Retry arrives (same request)

API:

1. Find key `abc123`
    
2. Compare hash:
    

`stored: H1 incoming: H1`

Match →

3. Return stored response_json
    
4. **Do NOT create new payment**
    

---

#### Conflicting retry (bug or fraud)

`same key abc123 but amount = 5000`

Hash mismatch:

`stored H1 != new H2`

→ reject with error:

> "Idempotency key reused with different request"

---

## Why expiration time?

You **cannot store idempotency forever**:

- storage grows unbounded
    
- keys only useful for retry window
    

Industry practice:

- expire in 24h / 48h
    

After expiry:

- new request with same key treated as new
    

---

## Are idempotency keys created by server?

❌ No.

Clients create them.

Server enforces them.

This is critical.

---

## Do we create idempotency keys for every endpoint?

Yes, for **all non-idempotent operations**:

| Endpoint                      | Needs idempotency? |
| ----------------------------- | ------------------ |
| POST /payments                | ✅                  |
| POST /payments/{id}/authorize | ✅                  |
| POST /payments/{id}/capture   | ✅                  |
Because all of these mutate money state.

---
## Why payment state alone is not sufficient

Payment state prevents:

`authorize twice`

But it does NOT prevent:

`create twice`

And it does NOT handle:

- network retry before state persisted
    
- crash between DB write and response
    
- duplicate HTTP requests
    

Idempotency keys solve **transport-level duplication**.  
Payment state solves **business-level validity**.

You need both.

---

## Summary for idempotency

|Concern|Solved by|
|---|---|
|Network retries|Idempotency keys|
|Duplicate requests|Idempotency keys|
|Conflicting retries|request_hash|
|Invalid transitions|Payment state machine|

---

# 2️⃣ Kafka deduplication – processed_events table

Kafka guarantees:

> at least once delivery

Meaning:

- same message **can be delivered multiple times**
    

You must handle this.

---

## What is event_id?

Every event you publish includes:

```ini
event_id = UUID v4
```

Generated by producer (API or worker).

It uniquely identifies the **message**, not the payment.

---

## Example

Payment capture emits:

```json
{
  "event_id": "e1",
  "event_type": "PAYMENT_CAPTURED",
  "payment_id": "p1"
}
```
Kafka might deliver this:

- once
    
- twice
    
- five times
    

---

## Consumer logic with processed_events

Pseudocode:

```ts
onMessage(event):
  if exists(processed_events.event_id):
      return // already processed

  begin transaction:
    process business logic
    insert into processed_events(event_id)
  commit
```
---

## Why this works

Even if Kafka redelivers:

- second time → event_id already exists
    
- skip safely
    

---

## How this gives exactly-once–style semantics

You combine:

- DB transaction
    
- unique constraint on event_id
    

So:

| Scenario                     | Result          |
| ---------------------------- | --------------- |
| Worker crashes before commit | message retried |
| Worker crashes after commit  | dedupe skips    |
| Kafka duplicates message     | dedupe skips    |

This gives:

> Effectively-once processing

(not theoretical exactly-once, but production-grade)

### More elaboration on: Worker crashes before commit -> message retried ⤵️

> Yes. If the worker crashes **before the DB commit**, the message will be processed again.  
> And that is **correct and desired**.

This is why we call it:

> **effectively-once** or **exactly-once–style**, not true exactly-once.

Let me make this precise.

---

## Two cases

### Case 1: Crash **before commit**

Flow:

`process logic (write ledger / update payment) CRASH ❌ before DB commit`

Result:

- DB transaction is rolled back
    
- `processed_events` not written
    
- Kafka re-delivers message
    
- Worker processes again
    

Outcome:

> Business effect happens **once** (because first attempt was rolled back)

✅ Correct behavior.

---

### Case 2: Crash **after commit**

Flow:

`process logic write ledger insert processed_events COMMIT ✅ CRASH ❌`

Kafka re-delivers.

Worker:

`sees event_id already in processed_events skips`

Outcome:

> Business effect still happens **once**

✅ Correct behavior.

---
## Why this is the best you can do

True exactly-once across:

- Kafka
    
- Network
    
- DB
    
- Crashes
    

is **theoretically impossible** without special infrastructure (Kafka transactions + same DB, etc).

So production systems do this:

> at-least-once delivery + idempotent processing

Which guarantees:

> the _result_ is exactly once

That’s what matters.

---

## One-sentence summary

> Messages may be processed multiple times, but **effects are applied once**.

That is what companies mean by “exactly-once–style”.

---

## Why payment_id + state is NOT enough

Two different events can exist:

`PAYMENT_CAPTURED (retry) PAYMENT_CAPTURED (replayed from log)`

Same payment_id + state, but different delivery.

event_id distinguishes **messages**, not domain objects.

---

## DLQ connection

If processing fails repeatedly:

- send event to `payments.dlq`
    
- record event_id there
    
- manual reconciliation later

# 3️⃣ How this fits into YOUR system concretely

---

## API Server

For create / authorize / capture:

- requires client Idempotency-Key
    
- stores idempotency_keys
    
- emits event with new event_id
    

---

## Settlement Worker

For each PAYMENT_CAPTURED:

- check processed_events
    
- settle payment
    
- insert processed_events(event_id)
    
- emit PAYMENT_SETTLED with new event_id
    

---

## Ledger Worker

Same dedupe logic.

# 4️⃣ Mental model (important)

Two different kinds of duplication:

|Type|Solution|
|---|---|
|HTTP duplicate requests|Idempotency keys|
|Kafka duplicate messages|processed_events|

Both are mandatory in financial systems.


## DB schema: Deep Dive

```sql
-- Enable UUID generation

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

  

-- Payments table (workflow state)

CREATE TABLE payments (

id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

amount_minor BIGINT NOT NULL CHECK (amount_minor > 0),

currency CHAR(3) NOT NULL,

direction VARCHAR(3) NOT NULL CHECK (direction IN ('IN', 'OUT')),

state VARCHAR(16) NOT NULL CHECK (

state IN (

'CREATED',

'AUTHORIZED',

'CAPTURED',

'SETTLED',

'FAILED'

)

),

created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

updated_at TIMESTAMPTZ NOT NULL DEFAULT now()

);

  

-- Idempotency keys table

CREATE TABLE idempotency_keys (key TEXT PRIMARY KEY,);

  

-- Immutable ledger table

CREATE TABLE ledger_entries (

id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

payment_id UUID NOT NULL REFERENCES payments(id),

currency CHAR(3) NOT NULL,

amount_minor BIGINT NOT NULL CHECK (amount_minor > 0),

entry_type VARCHAR(6) NOT NULL CHECK (entry_type IN ('CREDIT', 'DEBIT')),

created_at TIMESTAMPTZ NOT NULL DEFAULT now()

);

  

-- Kafka consumer deduplication table

CREATE TABLE processed_events (

event_id UUID PRIMARY KEY,

processed_at TIMESTAMPTZ NOT NULL DEFAULT now()

);
```

This SQL file defines **the persistence model for your payments system**.  
Together, these tables support **workflow state**, **financial correctness**, **idempotency**, and **exactly-once event processing**.

I’ll explain it from the _why_ first, then table by table.

---

## Big picture

This schema is doing **four distinct jobs**:

1. **Store payment workflow state** (`payments`)
    
2. **Guarantee idempotent API behavior** (`idempotency_keys`)
    
3. **Record money movements immutably** (`ledger_entries`)
    
4. **Prevent double-processing of events** (`processed_events`)
    

This mirrors a **production-grade payments architecture**.

---

## 1. UUID support

`CREATE EXTENSION IF NOT EXISTS "uuid-ossp";`

This enables:

`uuid_generate_v4()`

Why:

- Avoids sequential IDs (safer, scalable)
    
- Works well across distributed systems
    
- Matches your TypeScript `id: string` model

## 2. `payments` — workflow & orchestration table

```sql
CREATE TABLE payments (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    amount_minor BIGINT NOT NULL CHECK (amount_minor > 0),
    currency CHAR(3) NOT NULL,
    direction VARCHAR(3) NOT NULL CHECK (direction IN ('IN', 'OUT')),
    state VARCHAR(16) NOT NULL CHECK (state IN ('CREATED', 'AUTHORIZED', 'CAPTURED', 'SETTLED', 'FAILED')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
### What this table represents

This table is **not a ledger**.  
It represents:

- the _current state_ of a payment
    
- its lifecycle position
    
- orchestration status
    

This maps directly to your `Payment` class.

---

### Column meanings

|Column|Purpose|
|---|---|
|`id`|Payment identifier|
|`amount_minor`|Amount in smallest unit (cents)|
|`currency`|ISO-4217 code|
|`direction`|IN / OUT|
|`state`|Payment lifecycle state|
|`created_at`|When payment was created|
|`updated_at`|Last state change|

---

### Important design choices

#### ✅ State constraints

`CHECK (state IN (...))`

This enforces **database-level invariants**:

- DB cannot store invalid states
    
- Matches `PaymentState` enum exactly
    

This prevents corruption even if application code fails.

---

#### ⚠️ Mutable by design

- Rows are updated as state changes
    
- This is intentional for _workflow tracking_
    
- **Not used for financial reporting**

## 3. ⭐ `idempotency_keys` — API safety

```sql
CREATE TABLE idempotency_keys (
    key TEXT PRIMARY KEY,
    request_hash TEXT NOT NULL,
    response_json JSONB NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at TIMESTAMPTZ NOT NULL
);
```
### What problem this solves

> _“What if the client retries the same request?”_

This table ensures:

- same request
    
- same response
    
- no duplicate side effects
    

---

### How it works conceptually

1. Client sends request with `Idempotency-Key`
    
2. Server hashes request payload
    
3. If key exists:
    
    - return stored `response_json`
        
4. If not:
    
    - process request
        
    - store response
        

This is **critical for payments**.

---

### Why store the response?

Because:

- You must return _exactly the same response_ ⭐⭐⭐⭐⭐
    
- Even across retries, crashes, or deploys
    

---

## 4. `ledger_entries` — source of financial truth

```sql
CREATE TABLE ledger_entries (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    payment_id UUID NOT NULL REFERENCES payments(id),
    currency CHAR(3) NOT NULL,
    amount_minor BIGINT NOT NULL CHECK (amount_minor > 0),
    entry_type VARCHAR(6) NOT NULL CHECK (entry_type IN ('CREDIT', 'DEBIT')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
### This is the most important table financially

This table is:

- **append-only**
    
- **immutable**
    
- **auditable**
    

Every movement of money results in **new rows**, never updates.

---

### Key concepts

|Concept|Explanation|
|---|---|
|`payment_id`|Links ledger entries to workflow|
|`CREDIT / DEBIT`|Accounting direction|
|`amount_minor`|Always positive|
|`created_at`|Ordering & audit|

---

### Why separate from `payments`?

Because:

- Workflow state can change
    
- Ledger history **must not**
    
- Auditors trust ledgers, not workflow tables
    

This is standard in banking systems.

---

## 5. `processed_events` — exactly-once processing

```sql
CREATE TABLE processed_events (
    event_id UUID PRIMARY KEY,
    processed_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
### What this solves

Kafka (and similar systems) are **at-least-once** delivery.

This table ensures:

- if an event is redelivered
    
- it is **not processed twice**
    

---

### Typical usage

```sql
INSERT INTO processed_events (event_id)
VALUES ($eventId)
ON CONFLICT DO NOTHING;
```
If the insert fails:

- event was already handled
    
- safely skip processing
